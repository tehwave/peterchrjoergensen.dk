## Idéen

Jeg så en på Reddit, der havde gjort sin GitHub-bidragsgraf til et brick breaker-spil. Jeg syntes, det var genialt. Spil og webudvikling, to ting jeg elsker, smækket sammen.

Jeg bygger hver eneste dag. Den grønne graf er et års arbejde presset sammen til firkanter. Det føltes rigtigt at gøre den spilbar. Mere end det: Det føltes _meningsfuldt_. Hver mursten er en dag, hvor jeg mødte op. At knuse dem er ikke bare tilfredsstillende. Det er at ødelægge beviset på min egen vedholdenhed på den sjovest mulige måde.

## Sådan virker det

Hver dag med bidrag bliver til en mursten. Farven og styrken følger GitHubs bidragsniveauer:

- **Lysegrøn** (1-3 bidrag) = 1 træf for at knuse den
- **Mellemgrøn** (4-6 bidrag) = 2 træf
- **Mørkegrøn** (7-9 bidrag) = 3 træf
- **Mørkeste grøn** (10+ bidrag) = 4 træf

Spillet henter bidragsdata fra begge mine GitHub-konti (`tehwave` og `peterchrjoergensen`) og lægger dem sammen. Hvis jeg lavede tre bidrag på den ene konto og fem på den anden, bliver dagen til én mursten med otte bidrag — sværere at knuse og flere point værd. Sammenlægningen sker under buildet, så spillet indlæses med det samme uden API-kald.

## Den tekniske stack

Det hele kører i én Astro-komponent. Under buildet henter frontmatter bidragsdata fra GitHubs GraphQL-API:

```typescript
const contributionData = await fetchMultipleContributions(profiles);
```

Dataene bliver bagt ind i en JSON-klump, som spillet i browseren læser fra attributten `data-bricks`. Ingen API-kald under kørsel. Spillet virker offline, når det først er indlæst.

Selve spillet er cirka 2.400 linjer almindelig JavaScript i et `<script>`-tag — ingen React og ingen spilframeworks. Kun Canvas API og `requestAnimationFrame`. Jeg ville have noget selvstændigt, der ikke pustede bundlen op.

### Spilloopet

Standardmønstret: `requestAnimationFrame` driver loopet og kalder `update()` og derefter `render()` for hver frame. Fysikken kører med skærmens opdateringsfrekvens, normalt 60 fps.

```javascript
function gameLoop() {
  if (gameState !== GameState.PLAYING) return;
  update();
  render();
  animationFrameId = requestAnimationFrame(gameLoop);
}
```

### Kollisionsregistrering

Til murstenene bruger jeg AABB-kollision — Axis-Aligned Bounding Box — hvor jeg kontrollerer, om boldens afgrænsning overlapper murstenens. Det er hurtigt og godt nok til rektangler.

Det svære er at finde ud af, _hvilken side_ bolden ramte. Retningen på opspringet afhænger af det. Jeg beregner indtrængningsdybden på alle fire sider og lader bolden hoppe fra den side, der har det mindste overlap:

```javascript
const minOverlapX = Math.min(overlapLeft, overlapRight);
const minOverlapY = Math.min(overlapTop, overlapBottom);

if (minOverlapX < minOverlapY) {
  ball.dx = -ball.dx; // Ramte venstre eller højre side
} else {
  ball.dy = -ball.dy; // Ramte toppen eller bunden
}
```

### Vinklen ved træf på battet

Stedet, hvor bolden rammer battet, bestemmer opspringsvinklen. Det giver spilleren kontrol:

- **Træf på venstre kant** → hopper op mod venstre, cirka 135°
- **Træf i midten** → hopper lige op, 90°
- **Træf på højre kant** → hopper op mod højre, cirka 45°

Matematikken oversætter træfpositionen fra 0 til 1 til et vinkelinterval på ±63° fra lodret:

```javascript
const hitPos = (ball.x - paddle.x) / paddle.width;
const angle = -Math.PI / 2 + (hitPos - 0.5) * Math.PI * 0.7;
```

## Effekterne, der får det til at føles godt

Jeg brugte det meste af tiden på den visuelle finish. Et brick breaker-spil uden saft føles dødt. Du har brug for det tilfredsstillende _dunk_, når bolden rammer battet, fornemmelsen af fart ved en serie af træf og panikken, når du er nede på dit sidste liv.

Det er ikke bare pynt. Det er forskellen på et spil, man prøver én gang, og et spil, der får en til at sige "bare ét forsøg mere".

### Bevægelsesspor

Når bolden bevæger sig hurtigere end grundfarten, tegner den udtonede kopier på sine tidligere positioner. For hver frame føjer jeg boldens position til en liste med højst otte punkter. Under rendering bliver ældre punkter mindre og mere gennemsigtige:

```javascript
ballTrail.forEach((pos, i) => {
  const opacity = (i / ballTrail.length) * 0.5;
  const size = ball.radius * (0.5 + (i / ballTrail.length) * 0.5);
  // Tegn en udtonet cirkel ved pos
});
```

Det skaber en komethale, der gør hurtige bolde lettere at følge.

### En aflang bold — squash og stretch

Over 1,5 gange grundfarten strækker bolden sig til en ellipse i bevægelsesretningen. Det er et klassisk animationsprincip: Hurtige objekter ser aflange ud.

Jeg bruger `atan2` til at finde hastighedens vinkel, roterer canvaset og tegner en ellipse i stedet for en cirkel:

```javascript
const angle = Math.atan2(ball.dy, ball.dx);
const elongation = Math.min(speed * 0.8, ball.radius * 2);

ctx.translate(ball.x, ball.y);
ctx.rotate(angle);
ctx.ellipse(0, 0, ball.radius + elongation, ball.radius, 0, 0, Math.PI * 2);
```

### Et vuggende bat

Når battet pludselig skifter retning, vugger det som gelé. Det er fjederfysik: Vugningen accelererer mod midten og dæmpes, så den falder til ro.

Battet skævvrides også med squash og stretch ud fra hastigheden. Bevægelse mod venstre trykker venstre side sammen, og bevægelse mod højre trykker højre side sammen. Effekten er diskret, men får battet til at føles fysisk i stedet for robotagtigt.

### Ildpartikler på sidste liv

Når du kun har ét hjerte tilbage, går der ild i battet. Trekantede partikler fra orange til rød opstår og stiger. Ingen HUD-advarsel er nødvendig. Ilden _er_ advarslen.

```javascript
if (lives === 1) {
  // Opret flammepartikler tæt på battet
  fireParticles.push({
    x: paddle.x + Math.random() * paddle.width,
    y: paddle.y - 5,
    vx: (Math.random() - 0.5) * 2,
    vy: -1 - Math.random() * 2,
    life: 1,
    size: 3 + Math.random() * 3,
  });
}
```

### Faldende sand — cellulær automat

Det er min yndlingseffekt. Knuste mursten forsvinder ikke bare. De eksploderer i partikler, der falder og lægger sig.

Jeg bruger en gitterbaseret cellulær automat i stedet for fysik på individuelle partikler. Hver celle indeholder enten farvet sand eller er tom. For hver frame behandler jeg gitteret nedefra og op:

1. Hvis cellen nedenunder er tom → fald ned
2. Ellers prøv at glide diagonalt, tilfældigt mod venstre eller højre
3. Hvis intet er åbent → bliv stående

```javascript
if (sandGrid[y + 1][x] === null) {
  sandGrid[y][x] = null;
  sandGrid[y + 1][x] = cell;
} else {
  const dir = Math.random() < 0.5 ? -1 : 1;
  if (sandGrid[y + 1][x + dir] === null) {
    sandGrid[y][x] = null;
    sandGrid[y + 1][x + dir] = cell;
  }
}
```

Behandlingen nedefra og op forhindrer partikler i at bevæge sig flere gange i samme frame. Gittertilgangen er O(celler) i stedet for O(n²) til kollisionskontroller — hundredvis af partikler uden mærkbar påvirkning af ydeevnen.

### Ekstra visuelle partikler — glimt

Oven på sandet tilføjede jeg små rent visuelle partikler, der toner ud og aldrig bliver en del af gitteret. De opstår ved træf på mursten, falder med tyngdekraften og forsvinder hurtigt — lige præcis nok glimt til at sælge sammenstødet uden rod. De er bittesmå med en radius på to pixels og alpha-udtoning, så træffet kan mærkes uden at gøre spillefeltet uklart.

## Point og progression

### Highscore

Jeg gemmer den bedste score lokalt og opdaterer den både ved sejr og nederlag. Det er bare `localStorage` med én nøgle, `brick-breaker-high-score`, som indlæses ved start og skrives, når den tidligere rekord bliver slået. Brugerfladen opdateres gennem `aria-live`, så skærmlæsere annoncerer ændringer, og overlayet viser "Ny highscore!", når man opnår den.

### Power-up: dagens mursten

GitHub viser dagens firkant. Jeg giver den en orange accent og ét liv, og når den knuses, udløser den en kort power-up. I ti sekunder fordobles pointene, og canvaset får en diskret animeret kant med kromatisk aberration — røde, grønne og blå streger forskudt med sinusbølger, så det føles levende uden at larme.

## Beslutninger om responsivt design

Spillet tilpasser sig skærmstørrelsen. Canvasbredden er begrænset til 320-1.000 pixels. Murstenenes størrelse skalerer proportionalt; de er altid kvadratiske ligesom GitHubs graf.

På små skærme — telefoner i portrættilstand — viser spillet beskeden "drej din enhed" og sætter spillet på pause. Registreringen er enkel:

```javascript
function shouldShowRotateMessage() {
  return isSmallDevice() && isPortrait();
}
```

En lille enhed er alt under 500 pixels i viewport-højde eller 640 pixels i bredde. Portrættilstand er, når højden er større end bredden. Spillet stopper automatisk, når beskeden vises, og fortsætter, når enheden drejes.

## Bygget med AI

Jeg lavede det på én onsdag aften. Jeg skrev næsten ingen kode i hånden. I stedet parprogrammerede jeg med Claude Sonnet 4.5 gennem GitHub Copilot i VS Code.

Min arbejdsgang:

1. **Beskriv det, jeg vil have** — "Tilføj et bevægelsesspor til bolden, når den bevæger sig hurtigt"
2. **Gennemgå implementeringen** — Læs den genererede kode, og forstå tilgangen
3. **Test og giv feedback** — "Sporet er for langt og føles forsinket. Gør det kortere, og lad det tone hurtigere ud"
4. **Iterér** — Gentag, indtil det føles rigtigt

AI'en håndterede detaljerne i Canvas API'et, matematikken til kollisioner og partikelsystemerne. Jeg fokuserede på spilfølelse: test, justering af konstanter og afvisning af tilgange, der så forkerte ud.

Nogle ting krævede flere forsøg. Battets vuggen skulle gennem nogle iterationer, før fjederfysikken føltes hoppende uden at flimre. Sandsimuleringen skulle justeres, før den så naturlig og ikke mekanisk ud.

Da kernemekanikken virkede, bad jeg min [specialbyggede Documentator-agent](https://github.com/tehwave/peterchrjoergensen.dk/blob/master/.github/agents/documentator.agent.md) om at gennemgå og forklare, _hvorfor_ hver beslutning var truffet. Matematikken for battets kollisionsvinkel er forklaret i detaljer. Sandgitteret forklarer, hvorfor vi behandler nedefra og op. Fjederfysikken bag vugningen er kommenteret med begrundelsen.

Resultatet er cirka 2.400 linjer, jeg faktisk kan vedligeholde, fordi jeg forstår hver linje, selv om jeg ikke selv skrev de fleste. Mit fremtidige jeg vil takke mit tidligere jeg for forklaringerne.

## Optimering af ydeevnen

Et spil, der animerer 60 gange i sekundet, skal passe på spildt arbejde. Her er det, jeg gjorde for at holde det hurtigt.

### Intersection Observer

Spillet stopper automatisk, når man scroller forbi det. Der er ingen grund til at køre fysik og rendere frames, som ingen ser:

```javascript
intersectionObserver = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (!entry.isIntersecting && gameState === GameState.PLAYING) {
      pauseGame();
    }
  });
});
```

Når canvaset forlader viewporten, stopper spilloopet. Scroll tilbage, og fortsæt, hvor du slap. Det er især vigtigt på lange sider, hvor spillet ligger i en sektion. Der er ingen grund til at brænde CPU-cyklusser af, mens brugeren læser andet indhold.

### Page Visibility API

Samme princip gælder browserfaner. Skift fane, og spillet stopper:

```javascript
document.addEventListener("visibilitychange", () => {
  if (document.hidden && gameState === GameState.PLAYING) {
    pauseGame();
  }
});
```

Det forhindrer spillet i at køre i baggrunden og spise batteri på mobilen.

### Debounced resize-handler

Resize-events på vinduet affyres konstant, mens man trækker. Det er spild at genberegne murstenenes position i hver eneste frame under en ændring af størrelsen:

```javascript
handleResize = debounce(resizeCanvasHandler, 100);
```

I stedet venter jeg 100 millisekunder efter den sidste resize-event, før jeg beregner igen. Trækket er jævnt uden hak, og gitteret bliver opdateret i det øjeblik, man stopper.

### Cellulær automat til partikler

Det faldende sand bruger en gitterbaseret cellulær automat i stedet for individuelle partikelobjekter med fysik. Hver celle er enten tom eller indeholder sand. Behandlingen af gitteret er O(celler): ét gennemløb for hver frame.

Individuelle partikler ville være O(n²) til kollisionskontroller. Hver partikel mod hver anden. Med hundredvis af partikler fra eksploderende mursten bliver det tusindvis af kontroller for hver frame. Gitteret håndterer det uden problemer.

### Datahentning under build

GitHub-bidragene hentes én gang under buildet og ikke ved hver sideindlæsning:

```typescript
const contributionData = await fetchMultipleContributions(profiles);
```

Dataene bliver bagt ind i en JSON-klump i HTML'en. Ingen API-kald under kørsel. Spillet virker offline. Første rendering er øjeblikkelig.

## Det ville jeg ændre

Spillet er ikke perfekt:

- **Ingen lyd** — Lyd ville tilføre meget saft: knuste mursten, træf på battet og musik ved game over. Lige nu mangler halvdelen af feedbackloopet.
- **Touch på mobil kunne være blødere** — Touchinput virker, men er ikke lige så responsivt som musen. Den taktile forbindelse er ikke helt på plads endnu.
- **Sværhedsgraden** — Dage med mange bidrag og mursten med fire liv er måske for svære. Jeg har ikke testet nok til at vide, om succes føles opnåelig eller frustrerende.

Men det virker. Og det viser noget virkeligt: Jeg møder op og bygger hver dag. Nu kan du spille dig gennem den vedholdenhed.
